# namaka

> Snapshot testing tool for the Nix language, built on haumea. Write expressions without hardcoding expected values -- namaka captures outputs as snapshot files for interactive review.

Source: `github:nix-community/namaka`

## Purpose

namaka eliminates manual reference value maintenance in Nix tests. Instead of writing `assert foo == bar;`, you define the expression `foo` and namaka manages snapshot files containing expected outputs. When outputs change, an interactive review interface lets you accept or reject updates. Inspired by Rust's `insta` snapshot testing library.

______________________________________________________________________

### Quick Start

Source: https://github.com/nix-community/namaka

Initialize a new project and run the testing workflow:

```console
nix flake init -t github:nix-community/namaka
nix develop

namaka check    # run checks, prepare snapshots for failures
namaka review   # interactively approve or reject pending snapshots
```

### CLI Commands

Source: https://github.com/nix-community/namaka

| Command | Description |
|---------|-------------|
| `namaka check` | Wrapper around `nix flake check`; prepares snapshots for failed tests |
| `namaka review` | Interactively accept or reject pending snapshots |
| `namaka clean` | Remove unused and pending snapshots |

**Flags:**

| Flag | Description |
|------|-------------|
| `-c, --cmd` | Custom command to run instead of defaults |
| `-d, --dir` | Change working directory |
| `-V, --version` | Print version |

### Test Structure

Source: https://github.com/nix-community/namaka

Tests follow a directory convention. Each test case is a subdirectory containing an expression file and an optional format file:

```
tests/
  test-name/
    expr.nix       # (required) the expression to test
    format.nix     # (optional) serialization format
```

**`expr.nix`** -- Contains the Nix expression under test. The evaluated result is compared against the stored snapshot.

**`format.nix`** -- Controls how the expression result is serialized. Three options:

| Format | Serializer | Description |
|--------|-----------|-------------|
| `"json"` | `builtins.toJSON` | Default format |
| `"pretty"` | `lib.generators.toPretty` | Human-readable Nix value |
| `"string"` | identity | Output as-is (must be a string) |

```nix
# format.nix
"pretty"
```

### Configuration -- namaka.toml

Source: https://github.com/nix-community/namaka

Place `namaka.toml` in your project root to customize behavior:

```toml
dir = "."

[check]
cmd = "nix flake check --no-build"

[eval]
cmd = "nix eval .#checks"
```

| Key | Description |
|-----|-------------|
| `dir` | Working directory |
| `[check].cmd` | Command run during `namaka check` (default: `nix flake check`) |
| `[eval].cmd` | Command for `namaka review` and `namaka clean` |

### Integration with Flakes

Source: https://github.com/nix-community/namaka

namaka uses `haumea.lib.load` to discover and load tests from the test directory. The library is exposed as a flake output, and checks are generated by loading test directories with the namaka library injected.

```nix
{
  inputs = {
    namaka.url = "github:nix-community/namaka";
    haumea.url = "github:nix-community/haumea/v0.2.2";
  };

  outputs = { self, namaka, haumea, nixpkgs, ... }:
    let
      lib = haumea.lib.load {
        src = ./nix;
        inputs = { inherit namaka; };
      };
    in {
      checks = lib.load {
        src = ./tests;
        inputs = { inherit (self) lib; };
      };
    };
}
```

### Snapshot Workflow

Source: https://github.com/nix-community/namaka

1. Write `expr.nix` with your test expression
2. Run `namaka check` -- test fails because no snapshot exists yet
3. Run `namaka review` -- inspect the output and accept to create the snapshot
4. On subsequent runs, `namaka check` compares against stored snapshots
5. When outputs change, `namaka review` shows diffs for approval

## Use Cases

- **Library testing**: Validate that Nix library functions produce expected attribute sets
- **Module testing**: Snapshot NixOS module evaluation results
- **Regression detection**: Catch unintended output changes across refactors
- **CI integration**: `namaka check` in CI, `namaka review` locally to approve changes
